## 良い点 (Strengths)

v0.1.0 から維持されている設計上の防御線をそのまま踏襲。本リリースでも以下が継続して有効です。

1. **`process.exit` を直接呼ばない bin 設計** — `bin/mme.ts` は `process.exitCode` を立てるだけ。`uncaughtException` / `unhandledRejection` のセーフ ネット付き。
2. **`writeAtomic` の堅牢な実装** — 同一ディレクトリーの `tmp + rename`、`randomBytes(6)` で衝突しない suffix、失敗時の best-effort tmp 削除。
3. **入力 JSON / 引数の strict 検証** — `validateTagPayload` / `validateChapters` / `parseChaptersJson` で coerce なしの厳格検証。
4. **stdout / stderr の責務分離** — Logger 経由でエラー・警告は必ず stderr、stdout は機械可読 JSON か値・バイナリのみ。
5. **依存関係が `commander` と core の 2 つだけ** — 1.0.0 でも runtime `dependencies` は `@akabeko/music-metadata-editor` と `commander ^12.1.0` のみ。サプライ チェーン リスク小。
6. **テスト経路と本番経路の統一** — `bin/mme.ts` と `runCli` (テスト) が同じ `createProgram` を共有。
7. **エラー → 終了コード マップが疎** — `exitCodeForMmeError` テーブルが意図しないコードを `Failure (1)` にフォールバック。
8. **色制御が一元化** — `output/colors.ts` の `resolveColorMode` が `FORCE_COLOR` / `--no-color` / `NO_COLOR` / TTY を 1 箇所で判定。
9. **チャプターの `id` 重複拒否** — `validateChapters` で `Set<string>` 重複検出。
10. **publish 構成の最小化** — `files: ["dist", "!dist/**/*.tsbuildinfo"]` の allow-list 構成で、`.tsbuildinfo` / sourcemap / fixtures / テスト / `node_modules` / `.env` の漏出を構造的に防止。`npm pack --dry-run` で 118 ファイル / 65.6 kB、漏出なしを実機確認済み。
11. **`workspace:^` による semver 範囲指定** — publish 時に `^1.0.0` に書き換わり、core のセキュリティ パッチ (1.x.y) を CLI 利用者へ自動伝播できる構成。同一作者の first-party 依存のため、サプライ チェーン リスクは npm 2FA 側で担保される範囲で十分。

## v0.1.0 から継続している所見

差分には含まれないが、v1.0.0 リリース時点でも以下は未対応です。stable 1.0.0 として公開しても致命的ではありませんが、1.x マイナーで順次解消することを推奨します。

### MED-1 — `--tag-file` / `--json` / `--text` / `--lrc` のサイズ上限なし

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium |
| 該当 | `commands/write/loadTagFile.ts`, `commands/chapter/parseChaptersJson.ts`, `commands/lyrics/loadLyricsInput.ts`, `commands/read/collectStdin.ts` |
| 状態 | **未対応** (v0.1.0 から継続) |

**所見.** いずれの入力 helper も `readFile` / `collectStdin` で全バイトをメモリに展開し、デコード後に `JSON.parse` 等を呼ぶ。`collectStdin` は単純なチャンク連結で上限なし。

**推奨.** 各入力の用途に応じた上限 (例: tag/JSON は 1 MB、LRC は 4 MB、stdin 全般は 64 MB) を helper に組み込み、超過時は `CommanderError(2, ...)` で usage error 化。共通ヘルパー `output/limits.ts` を追加するのが整然。

### MED-2 — `mme picture set --input` のサイズ無制限

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium |
| 該当 | `commands/picture/setPicture.ts:90-97` (`readPictureBytes`) |
| 状態 | **未対応** (v0.1.0 から継続) |

**所見.** `--input <path>` は `readFile` でファイル全体を読み、`--input -` は `collectStdin` で stdin 全体を読む。サイズ チェックなし。

**推奨.** picture には現実的な上限 (例: 32 MB) を設定し、超過時は `CommanderError(2)` で拒否。`--max-picture-bytes` でオーバーライド可能にすると柔軟。

### MED-3 — `writeAtomic` が元ファイルの permission を保持しない

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium |
| 該当 | `commands/write/writeAtomic.ts:20-29` |
| 状態 | **未対応** (v0.1.0 から継続) |

**所見.** `writeFile(tmp, bytes)` は既定モード (`0o666 & ~umask`) で tmp を作成し、`rename(tmp, path)` で元ファイルを置き換える。元ファイルの mode は引き継がれない。共有環境で `0o600` のファイルを書き換えると `0o644` 既定 (umask 依存) に変わり、意図せず他者が読めるようになる可能性。

**推奨.** 書き込み前に `stat(path)` で `mode` を取得し、`writeFile` 後に `chmod(tmp, mode & 0o7777)` を呼ぶ。元ファイルが存在しない (新規書き込み) ときは現状維持。

### MED-4 — `mme write --stdin --output -` が TTY 出力時のチェックなし

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium (UX 寄り) |
| 該当 | `commands/write/handleWrite.ts:129-130` (`runStreamMode`) |
| 状態 | **未対応** (v0.1.0 から継続) |

**所見.** stream mode で `--output -` 指定時、stdout が TTY でも問答無用で書き込む。バイナリで端末状態が破壊される可能性。直接的脆弱性ではないが CLI ハードニング項目として標準的。

**推奨.** `process.stdout.isTTY === true` のとき `CommanderError(2, ...)` で拒否。escape hatch として `--allow-tty-binary` 等を後付け。

### LOW-1 — エラー メッセージへのファイル パス露出

| 項目 | 値 |
| --- | --- |
| 重要度 | Low |
| 該当 | `errors/formatMmeError.ts:80-85`, `commands/write/writeAtomic.ts` の throw 経路 |
| 状態 | **未対応** (v0.1.0 から継続) |

**所見.** `writeAtomic` の rename / write 失敗エラーは Node の生 `Error` が `formatMmeError` を通って stderr に出る。tmp ファイルのフル パス (例: `/private/var/.../.foo.mp3.<hex>.tmp`) を含む可能性があり、CI ログに残る運用では情報量が増える。

**推奨.** `writeAtomic` 内で I/O 失敗を `Error("failed to atomically write \"<path>\": <syscall>")` のように丸めて再 throw。元の `cause` は保持しつつメッセージは整形版に。

### LOW-2 — シンボリック リンク経由の上書き

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `commands/write/handleWrite.ts` (`runFileMode`)、`commands/picture/setPicture.ts` の `loadTrack(args.file)` 経路全般 |
| 状態 | **未対応** (v0.1.0 から継続) |

**所見.** `mme write song.mp3` が `song.mp3` がシンボリック リンクの場合、リンク先を follow して上書きする。ローカル CLI として一般的な挙動だが、untrusted ディレクトリーで実行する運用では意外な書き換えが起きる。

**推奨.** デフォルト挙動は維持しつつ将来 `--no-follow-symlinks` を提供。短期では README に注記を追加する程度で十分。

### LOW-3 — `IO_ERROR_CODES` のカバレッジ

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `errors/formatMmeError.ts:26-35` |
| 状態 | **未対応** (v0.1.0 から継続) |

**所見.** `EROFS` / `EBUSY` / `EMFILE` などはカバー済みだが、`ENOSPC` / `ELOOP` / `ETIMEDOUT` / `EXDEV` は含まれず `Failure (1)` にフォールバック。

**推奨.** 1.x 期間に追加するか方針を決め、決めたら `plan/` の終了コード表と README の表も同時更新。
