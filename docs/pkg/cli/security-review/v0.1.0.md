## サマリー

致命的な脆弱性は **検出されず**。`writeAtomic` の `tmp + rename`、`process.exit` を直接使わない bin 設計、`MmeError` を疎マップで扱う `formatMmeError`、`validateChapters` / `validateTagPayload` の strict 検証など、CLI として求められる基本的な防御は揃っています。`v0.1.0` (リリース候補初版) として通常運用に支障となるリスクはなし。

|  | 件数 |
| --- | --- |
| Critical | 0 |
| High | 0 |
| Medium | 4 |
| Low / Informational | 3 |

Medium はすべて「リソース上限・モード保持・TTY 検出など 0.2 / 1.0 系に取り込みたい改善」レベル。

## 良い点 (Strengths)

1. **`process.exit` を直接呼ばない bin 設計** — `bin/mme.ts` は `process.exitCode` を立てるだけ。`uncaughtException` / `unhandledRejection` のセーフ ネット付き。Node ランタイムが pending I/O を flush し終えてから安全に終了。
2. **`writeAtomic` の堅牢な実装** — `commands/write/writeAtomic.ts` は同一ディレクトリーの `tmp + rename` で書き込み、cross-device rename の落とし穴を回避。`randomBytes(6)` で衝突しない suffix。失敗時は best-effort で tmp を削除。
3. **入力 JSON / 引数の strict 検証** —
   - `parseTagOverrides.validateTagPayload`: フィールド allowlist + 型 (string / integer / float[0,1]) を厳密に検証、coerce なし。
   - `validateChapters`: `id` 非空、`startMs` / `endMs` 有限、ordering、unique を全て assert。
   - `parseChaptersJson`: array / object 形状を事前検査。
4. **stdout / stderr の責務分離が明確** — 進捗・警告・エラーは `Logger` 経由で必ず stderr。stdout は機械可読 JSON か `--field` の値、または `--output -` のバイナリのみ。pipeline 連携で安全。
5. **依存関係が `commander` 1 つだけ** — `dependencies` は `commander ^12.1.0` と core の workspace 依存のみ。サプライ チェーン リスク小。
6. **テスト経路と本番経路の統一** — `runCli` (テスト用) と `bin/mme.ts` が同じ `createProgram` を共有。本番経路でだけ通る分岐がない。
7. **エラー → 終了コードのマップが疎** — `errors/exitCodes.ts` の `exitCodeForMmeError` テーブルは `unsupported-format` と `invalid-tag` のみ明示マップ、それ以外は `Failure (1)` にフォールバック。core が新コードを増やしても CLI 側で意図せず誤分類しない。
8. **色制御が一元化** — `output/colors.ts` の `resolveColorMode` が `FORCE_COLOR` / `--no-color` / `NO_COLOR` / TTY を 1 箇所で判定。コマンド実装が `process.env` を直接見ない。
9. **チャプターの `id` 重複拒否** — `validateChapters` で `Set<string>` 重複検出。ID3v2 の CHAP/CTOC 仕様上の前提を CLI 入口で守る。

## 所見 (Findings)

### MED-1 — `--tag-file` / `--json` / `--text` / `--lrc` のサイズ上限なし

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium |
| 該当 | `commands/write/loadTagFile.ts`, `commands/chapter/parseChaptersJson.ts`, `commands/lyrics/loadLyricsInput.ts`, `commands/read/collectStdin.ts` |
| 攻撃シナリオ | 巨大 JSON / LRC を `--tag-file -` 経由で渡し OOM を狙う。CLI 用途では現実的に小さいが、運用側で外部入力を pipe で繋ぐ場合の保護がない |

**所見.** いずれの入力 helper も `readFile` / `collectStdin` で **全バイトをメモリに展開** し、デコード後に `JSON.parse` 等を呼びます。`collectStdin` は単純なチャンク連結で上限なし。

**推奨.** 各入力の用途に応じた上限 (例: tag/JSON は 1 MB、LRC は 4 MB、stdin 全般は 64 MB) を `commands/.../*.ts` の helper に組み込み、超過時は `CommanderError(2, ...)` で usage error 化。共通ヘルパーとして `output/limits.ts` を追加するのが整然。

### MED-2 — `mme picture set --input` のサイズ無制限

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium |
| 該当 | `commands/picture/setPicture.ts:90-97` (`readPictureBytes`) |
| 攻撃シナリオ | 数 GB の画像を渡されると `Uint8Array` に展開した時点で OOM。画像をそのままタグへ埋め込むため、書き込み先メディアにも影響 |

**所見.** `--input <path>` は `readFile` でファイル全体を読み、`--input -` は `collectStdin` で stdin 全体を読みます。サイズ チェックなし。

**推奨.** picture には現実的な上限 (例: 32 MB) を設定し、超過時は `CommanderError(2)` で拒否。タグ自体のサイズは format ごとに上限が違うため、CLI 既定値 + `--max-picture-bytes` でオーバーライド可能にする方針が柔軟。

### MED-3 — `writeAtomic` が元ファイルの permission を保持しない

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium |
| 該当 | `commands/write/writeAtomic.ts:20-29` |
| 影響 | 共有環境 / サーバー運用時、`0o600` (本人のみ) のファイルを書き換えると `0o644` 既定 (umask 依存) に変わり、意図せず他者が読めるようになる可能性 |

**所見.** `writeFile(tmp, bytes)` は `fs.writeFile` の既定モード (`0o666 & ~umask`) で tmp を作成し、`rename(tmp, path)` で元ファイルを置き換えます。元ファイルの mode は引き継がれません。

**推奨.** 書き込み前に `stat(path)` で `mode` を取得し、`writeFile` 後に `chmod(tmp, mode & 0o7777)` を呼ぶ。元ファイルが存在しない (新規書き込み) ときは現状維持。owner / group の保持は Node.js `fs.chown` でも可能だが root 必要なので、permission のみ保持する範囲が現実的。

### MED-4 — `mme write --stdin --output -` が TTY 出力時のチェックなし

| 項目 | 値 |
| --- | --- |
| 重要度 | Medium (UX 寄り) |
| 該当 | `commands/write/handleWrite.ts:129-130` (`runStreamMode`) |
| 影響 | バイナリを TTY に流すとターミナル制御コードを誤って評価してターミナル状態を破壊する可能性 |

**所見.** stream mode で `--output -` 指定時、stdout が TTY でも問答無用で書き込みます。シェル ユーザーが手で叩いたとき、ターミナルがバイナリで壊れる事故が起きます。直接的な脆弱性ではないが、CLI ハードニング項目として標準的。

**推奨.** `process.stdout.isTTY === true` の場合は `CommanderError(2, ...)` で「`--output -` to TTY refused」を返す。`FORCE_COLOR=1` のような escape hatch (`--allow-tty-binary` 等) を後付けで提供。

### LOW-1 — エラー メッセージへのファイル パス露出

| 項目 | 値 |
| --- | --- |
| 重要度 | Low |
| 該当 | `errors/formatMmeError.ts:80-85`, `commands/write/writeAtomic.ts` の throw 経路 |

**所見.** `writeAtomic` の rename / write 失敗エラーは Node の生 `Error` が `formatMmeError` を通って stderr に出ます。tmp ファイルのフル パス (例: `/private/var/.../.foo.mp3.<hex>.tmp`) を含む可能性があり、CI ログに残る運用では情報量が増えます。

**推奨.** `writeAtomic` 内で I/O 失敗を `Error("failed to atomically write \"<path>\": <syscall>")` のように丸めて再 throw する。元の `cause` は保持しつつメッセージは整形版に。

### LOW-2 — シンボリック リンク経由の上書き

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `commands/write/handleWrite.ts` (`runFileMode`)、`commands/picture/setPicture.ts` の `loadTrack(args.file)` 経路全般 |

**所見.** `mme write song.mp3` が `song.mp3` がシンボリック リンクの場合、リンク先を follow して上書きする。一般的なローカル ツールの想定挙動と整合だが、untrusted ディレクトリーで実行する運用では意外な書き換えが起きる。

**推奨.** デフォルト挙動は維持しつつ、将来 `--no-follow-symlinks` (lstat → ELOOP 相当) を提供。短期では README に注記を追加する程度で十分。

### LOW-3 — `IO_ERROR_CODES` のカバレッジ

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `errors/formatMmeError.ts:26-35` |

**所見.** `EROFS` / `EBUSY` / `EMFILE` などはカバー済みだが、`ENOSPC` (容量不足) / `ELOOP` (シンボリック リンク無限ループ) / `ETIMEDOUT` などは含まれていない。これらは `Failure (1)` にフォールバック。終了コードの疎マップ方針と整合する選択ではあるが、運用観点で I/O ファミリーに足したい候補が残っている。

**推奨.** `0.x` 期間に `ENOSPC` / `ELOOP` / `ETIMEDOUT` / `EXDEV` を追加するか方針を決める。決めたら `plan/phase-01-foundation.md` の終了コード表と README の表も同時更新。
