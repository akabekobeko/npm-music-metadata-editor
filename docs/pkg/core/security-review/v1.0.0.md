## 良い点 (Strengths)

1. **`BufferCursor` の境界チェックが網羅的** — `packages/core/src/io/bufferCursor/bufferCursor.ts` の全 `readXxx` メソッドが `checkAvailable(need)` を呼び、過読を `RangeError` で必ず拒否。`readBytes` / `peek` / `seek` / `skip` は `Number.isInteger` も検証しているため、不正な長さや負のオフセット経由のアンダーフローが起きない。
2. **format detection が signature 優先** — `formats/detect.ts` は `detectFormatBySignature` 結果を最優先し、拡張子は signature 不一致時のみ参照する。renamed file による誤判定対策がコメントとともに明記。
3. **`MmeError` での明示的な失敗分類** — `errors/mmeError.ts` の `MmeErrorCode` (`unsupported-format` / `invalid-tag` / `truncated-input` / `unsupported-feature`) で失敗カテゴリを公開。`readMetadata` / `writeMetadata` は format reader/writer の plain `Error` を `invalid-tag` にラップし、`MmeError` であればそのまま再 throw する 2 段構えで、内部実装が漏出しない。
4. **不変モデル + Plain Object** — `types.ts` の `Track` / `TagData` などすべて `readonly`。`saveTrack` も `track` を受け取るだけで内部 mutation がない。意図しない side-effect を構造的に阻止。
5. **依存関係ゼロ (ランタイム)** — `package.json` の `dependencies` は空。サプライ チェーン由来の脆弱性面が極小。
6. **書き込み API がソース バイト列を mutate しない契約** — `FormatWriter` の JSDoc に `must not be mutated` と明記。`writeMetadata` は `loadInput` 経由の bytes をハンドラーに渡すだけで、戻り値の `Uint8Array` も新規バッファ。
7. **Plain Object モデル + factory function** — `MmeError` も class ではなく `createMmeError` で生成。`isMmeError` は `name === "MmeError"` の dual check で `instanceof` 跨ぎ realm でも安定。

## 所見 (Findings)

### LOW-1 — `readFileBuffer` がシンボリック リンクを follow する

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/core/src/io/file.ts:12` (`readFileBuffer`) |
| 攻撃シナリオ | core が untrusted パスを受け取るシナリオで、シンボリック リンクを差し替えて任意ファイルを読み込ませる TOCTOU 系の攻撃 |

**所見.** `readFileBuffer` は `node:fs/promises` の `readFile(path)` を直接呼んでおり、シンボリック リンクは透過的に follow されます。ライブラリーの想定ユース (利用者自身がパスを管理) では問題になりませんが、「上位アプリがエンド ユーザーから受け取ったパスをそのまま `loadTrack(path)` に渡す」というケースを想定すると、リンク経由の任意ファイル読み込みリスクが残ります。

**推奨.** ライブラリー API の契約として「path は呼び出し元が信頼済みであること」を README / `architecture.md` で明文化する (現状の架構と整合)。または将来 `ReadOptions.followSymlinks?: boolean` のような opt-out を提供する。

### LOW-2 — 入力サイズの上限がライブラリー側にない

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/core/src/api/loadInput.ts` / `readMetadata.ts` / `writeMetadata.ts` |
| 攻撃シナリオ | 巨大ファイルを介したメモリ枯渇 DoS (上位アプリで入力サイズを検証していない場合) |

**所見.** `loadInput` はファイル全体を `Uint8Array` に展開し、reader/writer も全バイトを in-memory に保持します。README / `architecture.md` でストリーミングは非ゴールと宣言されていますが、ライブラリー API には上限オプションがありません。

**推奨.** 短期的には「上位層 (CLI / GUI) でサイズ チェックする」ことを契約として明示。中期的には `ReadOptions.maxInputBytes?: number` のような上限を追加し、超過時に `MmeError("truncated-input")` か新コード `"input-too-large"` を投げる方向を検討。

### LOW-3 — `RangeError` が `invalid-tag` に丸められ、`truncated-input` 区別が失われる

| 項目 | 値 |
| --- | --- |
| 重要度 | Low |
| 該当 | `packages/core/src/api/readMetadata.ts:34-45` (catch ブロック) |
| 影響 | エラー カテゴリがより細かく取れる場面で `invalid-tag` に丸められ、上位の終了コード判定が粗くなる |

**所見.** `BufferCursor` の `RangeError` (= 入力末尾を超過) は `MmeErrorCode` の `truncated-input` に相当しますが、現状は `readMetadata` / `writeMetadata` の catch で **すべて `invalid-tag`** にラップされます。`MmeErrorCode` には `truncated-input` が定義済みのため、本来はここで分岐する余地があります。

**推奨.** v1 互換性を維持しつつ、各 format reader 内で `RangeError` を `createMmeError({ code: "truncated-input", cause })` に変換する補助を入れる。CLI 側 (`exitCodeForMmeError`) も将来的に `"truncated-input" → 5 (InvalidTag)` か新コードへマップ。

### LOW-4 — Picture `mimeType` の検証は呼び出し側責務

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/core/src/extras/picture/detectMime.ts` ほか各 converter |
| 影響 | 不正な MIME 文字列を含む `PictureInfo` をそのまま埋め込めるため、後段 (例: ブラウザー描画) で content-type confusion を誘発する可能性 |

**所見.** `detectMime` は known signature にマッチしないと `undefined` を返すだけで、ライブラリー全体としては **MIME 値が任意文字列** として扱われます。round-trip 互換性 (元ファイルの非標準 MIME を保持する) を優先した設計判断として妥当ですが、untrusted な `PictureInfo` を取り込んで再書き込みするケースでは、`mimeType` を再検証してから埋め込みたいケースもあります。

**推奨.** ライブラリーは現状の振る舞いを維持しつつ、`detectMime` を export して呼び出し元が二段検証できるようにする (現状は internal)。長期的には `WriteOptions.pictures` に対して任意の validator を渡すフックを検討。

### LOW-5 — `decodeSyncSafeInt32` の MSB 入力検証 (要追加調査)

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/core/src/utils/syncSafeInt/decodeSyncSafeInt32.ts` (本レビューでは個別未確認) |

**所見.** ID3v2 の syncsafe int は各バイトの最上位ビットが 0 である前提です。仕様外の入力に対する振る舞い (例: MSB が 1 のとき silently mask するか、エラーにするか) は呼び出しコンテキストに影響します。`BufferCursor.readSyncSafeInt32` は `decodeSyncSafeInt32(source, position)` を委譲呼び出ししており、MSB チェックの場所が散らばっていないことを次回更新時に確認したい。

**推奨.** 次回 review (v1.1 以降) で `decodeSyncSafeInt32.ts` の実装を読み、MSB 不正入力に対する挙動を確認・記録する。

### LOW-6 — `rating` 値の範囲検証は format writer 任せ

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/core/src/types.ts` (`TagData.rating`) |

**所見.** `TagData.rating` は `[0, 1]` 正規化済みの想定 (JSDoc に明記) ですが、core API レベルの範囲チェックはありません。CLI 側の `parseTagOverrides` ではチェック済みですが、core を直接呼ぶ利用者が `1.5` などを渡すと、format ごとに挙動が変わる可能性があります。

**推奨.** `writeMetadata` の入口で範囲外を `invalid-tag` で拒否するか、各 format writer 内で clamp を統一する。後者の場合は仕様変更となるため、まずは API 入口での fail-fast を推奨。
