## サマリー

致命的な脆弱性は **検出されず**。Phase 7 の差分は以下の方向で **セキュリティ的に改善**:

- v0.0.0 LOW-1 として持ち越していた **`webPreferences.sandbox: true` を本 PR で追加**。Renderer 侵害時の最終防衛ラインを 1 段強化。
- 新規 IPC (`mme:dialog:expandPaths` / `mme:menu:setState` / `mme:menu:action` / `mme:log:forward` / `mme:fatal` / `mme:fatal:report`) はいずれも **既存の `IpcResult<T>` + `toIpcError` パターンを踏襲**。`Error` インスタンスは IPC 越境せず、構造化クローンで剥がされる plain payload のみが流れる。
- D&D 展開は **`lstat` を使い symlink を辿らない / 深さ 3 制限 / 拡張子 allowlist** の三重防御で、悪意あるドロップによる任意ファイル列挙を構造的に塞いでいる。
- ネイティブ メニューの `action` ディスパッチは **閉じた string union** (`MenuAction`) のみを許可。`shell.openExternal` の URL は `main.ts` 内のハード コードで、Renderer から差し替え不可。
- 設定 patch (`theme` / `locale`) の追加でも、**`mergeSettings` の明示フィールド戦略**は維持されている (`patch.theme` / `patch.locale` の値だけを取り出して再構築)。spread や汎用 deep-merge は導入されていない。

## 良い点 (Strengths)

v0.0.0 から維持されている設計上の防御線をそのまま踏襲。Phase 7 で追加された箇所も以下の理由で安全:

1. **`webPreferences.sandbox: true` を追加** — `packages/gui/src/main/main.ts` で BrowserWindow に `sandbox: true` を明示。`contextIsolation: true` / `nodeIntegration: false` と合わせ、Chromium のサンドボックス内で Renderer が侵害されても Main の Node 権限に直接到達できない。preload (`.cjs` ビルド) は `electron` モジュールのみ参照しており、sandbox 制約と両立する。
2. **`expandDroppedPaths` の三重防御** — `packages/gui/src/main/dnd/expandDroppedPaths.ts` は (a) `lstat` で stat し `isSymbolicLink()` を弾く、(b) 深さ 3 階層で再帰打ち切り、(c) 拡張子 allowlist (`AUDIO_EXTENSIONS`) で出力をフィルター。仮に Renderer が侵害され任意の絶対パス (例: `/`) を投げても、結果として返るのは「実体ファイルかつ音楽拡張子」のみで、symlink 経由でユーザー権限の任意ディレクトリーに迂回することもできない。
3. **D&D の Path 解決は `webUtils.getPathForFile`** — `packages/gui/src/preload/preload.ts` で公開。Renderer が任意の File オブジェクトを構築してパスを入手することはできず、OS の drag 経由で渡された実体 `File` のみ resolve 可能 (DataTransfer 由来の File は OS から渡されるため Renderer が偽造できない)。
4. **IPC `MenuAction` が閉じた string union** — `packages/gui/src/main/ipc/types.ts` の `MenuAction` 型は固定 9 種。Main → Renderer の通知だけなので、Renderer が (Main を介さず) 任意 action を発火させる経路は無い。Renderer 受信側は `useMenuActions.ts` で `switch` の網羅 (`satisfies MenuAction`) を強制しており、未知 action は単に no-op。
5. **`mme:menu:setState` の payload は data のみ** — `packages/gui/src/main/ipc/onMenuSetState.ts` から `applyMenuStateSnapshot` を呼び、`{ ...controller.state, ...snapshot }` で 1 段のオブジェクト spread しか行わない。`__proto__` 等のキーは Electron の structured clone (V8 native) で strip され、かつ menu controller は label/checked/enabled のような Electron primitive 値しか参照しない。
6. **`shell.openExternal` の URL がハード コード** — `packages/gui/src/main/main.ts` の `PROJECT_WEBSITE_URL` 定数。`installAppMenu` の translate は `item.externalUrl` を click closure で capture して `shell.openExternal` に渡すが、`externalUrl` は `buildAppMenu` が `state.websiteUrl` から差し込むだけで、`state.websiteUrl` は `main.ts` の定数からのみ供給される。Renderer 由来の URL は流入経路が無い。
7. **`onLogForward` / `onFatalReport` は出力先がプロセス内のみ** — `console.*` 経由で `electron-log` のファイル / stdout 出力に行くだけで、Renderer 側に echo back しない。仮に Renderer が任意文字列を流しても、ログ ファイル (`<userData>/logs/main.log`) の閲覧はユーザー本人のみで、外部に漏れる経路が無い。レビュー方針 (HARD EXCLUSIONS #12 「ログ スプーフィング不問」) に該当。
8. **`setupFatalHandlers` の payload が plain object** — `packages/gui/src/main/fatal/setupFatalHandlers.ts` で `Error` を `{ source, message, stack? }` に詰め替えてから `webContents.send`。`Error.cause` などの隠れた循環参照や native chain は剥がれる。
9. **`mergeSettings` の明示フィールド戦略を維持** — `packages/gui/src/main/settings/mergeSettings.ts` で `theme` / `locale` を patch から取り出す際も、`patch.theme !== undefined` の条件でスカラ値だけを採用 (オブジェクト spread しない)。`sanitizeWidths` の数値フィルターも引き続き有効。
10. **`resolveLocale` の allowlist** — `packages/gui/src/main/locales/resolveLocale.ts` は preference / system locale を `SUPPORTED_LOCALES` (`["en", "ja"]`) と一致比較してから返す。永続化された値が将来別の値 (`"fr"` 等) に書き換わっていても安全に English にフォールバック。
11. **`electron-log` の標準ローテーション** — `setupLogger.ts` が `electron-log/main` の `initialize` を呼ぶだけで、ファイル ローテーション (1 MB) はライブラリ既定。任意のログ パスや transport を追加していないため、攻撃面は導入していない。
12. **`useDragAndDrop` が `event.preventDefault` を必ず呼ぶ** — `packages/gui/src/renderer/features/dnd/useDragAndDrop.ts` の `dragover` / `drop` ハンドラーは preventDefault を最初に呼ぶ。これがないと Chromium が dropped file を新ページとしてロードしてしまうが、本実装はその経路を塞いでいる。

## 所見 (Findings)

### 差分起因の新規 Finding

**該当なし**。Phase 7 で追加された経路は IPC 境界 / ファイル I/O / Electron 設定のいずれの観点からも新規脆弱性を導入していない。

### v0.0.0 から継続している LOW (本 PR で悪化なし)

- **LOW-2 — Content-Security-Policy が未設定 (継続)**: `packages/gui/src/renderer/index.html`。本 PR では未対処。React 経由で文字列はエスケープされるため XSS 経路は依然として無いが、二段目の防御として `default-src 'self'` ベースの CSP を `<meta>` か `onHeadersReceived` で適用するのが望ましい。Vite dev サーバーが `'unsafe-eval'` を要求するため production のみ適用する分岐が必要。
- **LOW-3 — クリップボード ペーストにセーフ ガードが無い (継続)**: `packages/gui/src/renderer/components/app/Spreadsheet/Spreadsheet.tsx` の `dispatchPaste`。本 PR では未対処。Save All で実ファイルへ書き戻されるため誤ペーストの影響は依然として残る。`applyPaste` 前段に「行数 > rows.length」「セル長 > N KiB」の確認ダイアログを挟むのが next step。
- **LOW-4 — `onSaveTrack` の許可リスト未実装 (継続 / Phase 7 で攻撃経路が増えた)**: `packages/gui/src/main/ipc/onSaveTrack.ts`。本 PR は **D&D (`mme:dialog:expandPaths`) と Open Recent (`mme:menu:action: openRecent`) を追加** したため、`filePath` を Main に渡す経路が以下 4 種に拡張された:
  1. `dialog.showOpenDialog` 由来 (v0.0.0 から)
  2. `recentFiles` (Phase 6 で追加)
  3. **D&D で `expandDroppedPaths` が返したパス (Phase 7 新規)**
  4. **Open Recent から再ロードしたパス (Phase 7 新規)**

  D&D 経由のパスは `expandDroppedPaths` が「実体ファイルかつ拡張子 allowlist 通過」であることを保証し、Open Recent も既存の `recentFiles` 集合からしか参照されないため、**新規 LOW として独立計上はしないが**、許可リスト実装時には上記 4 経路すべての成功パスを `Set<string>` (resolved) に追記する設計にする必要がある。
