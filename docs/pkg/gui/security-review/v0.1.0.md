## サマリー

致命的な脆弱性は **検出されず**。Phase 8 の差分はセキュリティ的に **中立 (新規攻撃面の追加なし)** かつ既存の LOW-3 (列ペーストのセーフ ガード未実装) を **構造的に部分緩和**:

- `ColumnDefinition.selectable: "cell-only"` の導入により、`Lyrics` / `Pictures` 列はヘッダー クリックで列選択状態に入らず、Cmd+V による列方向 broadcast の経路が型レベルで塞がれた。とくに **改行を含み得る `Lyrics` の全行上書き** が dispatch 段階でブロックされ、LOW-3 で懸念していた誤ペースト被害の最大ケースが成立しなくなる。
- 追加コードはすべて Renderer 内の純粋な UI ロジック (純関数 `isColumnSelectable` + ヘッダー クリック分岐 + `<th>` への `title` / `cursor-default` 付与) で、IPC / ファイル I/O / シェル / クリップボード書き込み / 動的コード実行のいずれの経路にも到達しない。
- `<th>` の `title` 属性に流れる文字列は `COLUMN_REGISTRY` (`constants.ts`) のハードコード文字列のみで、ユーザー入力は混入しない。React は属性値を自動エスケープするため `dangerouslySetInnerHTML` 系の懸念も無い。
- Phase 8 で **ランタイム依存の追加なし**。

## 良い点 (Strengths)

v0.0.0 → v0.1.0 (Phase 7) で確立した防御線をそのまま踏襲。Phase 8 で追加された箇所も以下の理由で安全:

1. **`webPreferences.sandbox: true` の維持** — `packages/gui/src/main/main.ts`。Phase 8 では Main / preload に変更が無く、Renderer サンドボックスの境界線は維持されている。
2. **`expandDroppedPaths` の三重防御 (継続)** — `packages/gui/src/main/dnd/expandDroppedPaths.ts` の `lstat` (symlink を辿らない) / 深さ 3 制限 / 拡張子 allowlist は引き続き有効。
3. **`webUtils.getPathForFile` 経由の D&D path 解決 (継続)** — `packages/gui/src/preload/preload.ts`。Renderer が任意 `File` を偽造して絶対パスを得る経路は依然として無い。
4. **IPC `MenuAction` の閉じた string union (継続)** — `packages/gui/src/main/ipc/types.ts`。Phase 8 では新規 IPC チャネル / action の追加なし。
5. **`mme:menu:setState` の payload が data のみ (継続)** — `applyMenuStateSnapshot` の単純 spread は `__proto__` を `electron` の structured clone で剥がす。
6. **`shell.openExternal` の URL がハードコード (継続)** — Renderer 由来の URL 流入経路は無い。
7. **ログ転送が出力先プロセス内のみ (継続)** — `onLogForward` / `onFatalReport` は外部 echo back しない。
8. **`setupFatalHandlers` の plain object payload (継続)** — `Error.cause` 等の native chain は剥がれる。
9. **`mergeSettings` の明示フィールド戦略 (継続)** — `theme` / `locale` は `patch.<field> !== undefined` でスカラ値のみ採用。
10. **`resolveLocale` の allowlist (継続)** — `SUPPORTED_LOCALES` (`["en", "ja"]`) との一致比較で safe fallback。
11. **ランタイム依存ゼロ ポリシー (継続)** — Phase 8 でも依存追加なし。Main は `electron` + `@akabeko/music-metadata-editor`、Renderer は React 系のみ。
12. **`useDragAndDrop` の `event.preventDefault` (継続)** — Chromium による dropped file の page navigation を引き続き抑止。
13. **`ColumnDefinition.selectable` 導入による列ペースト経路の構造的封じ (Phase 8 新規)** — `packages/gui/src/renderer/features/spreadsheet/types.ts` で `ColumnSelectability = "column" | "cell-only"` を定義し、`packages/gui/src/renderer/features/spreadsheet/constants.ts` で `lyrics` / `pictures` を `cell-only` 化。`packages/gui/src/renderer/components/app/Spreadsheet/Spreadsheet.tsx` の `handleColumnHeaderClick` は `isColumnSelectable(column)` が `false` の列で早期 return するため、`selection.kind === "column"` には決して入らない。続く `dispatchPaste` も `selection.kind === "none"` で短絡する。**結果として Lyrics / Pictures に対する列方向 broadcast 経路は型と実装の両面で閉じている**。
14. **`isColumnSelectable` が純関数 (Phase 8 新規)** — `packages/gui/src/renderer/features/spreadsheet/isColumnSelectable.ts`。`(column.selectable ?? "column") === "column"` の 1 行のみで I/O も副作用も無く、Object literal lookup と型チェックで網羅性が担保される。
15. **`<th>` 属性への文字列補間が信頼境界の内側のみ (Phase 8 新規)** — `<th>` の `title` 属性に流れる `${column.title} is editable per cell only — column-wide paste is disabled.` の `column.title` は `COLUMN_REGISTRY` ハードコード文字列由来で、外部入力は混入しない。React の属性自動エスケープと組み合わせ XSS 経路は成立しない。

## 所見 (Findings)

### Phase 8 差分起因の新規 Finding

**該当なし**。Phase 8 で追加されたのは Renderer 内の純粋な UI ロジック (型定義 / 純関数 / イベント ハンドラの分岐 / 属性付与) のみで、IPC / ファイル I/O / 動的コード実行 / 認証 / 暗号などの境界には触れない。むしろ既存の LOW-3 を部分緩和する方向に作用する。

### v0.1.0 から継続している LOW (Phase 8 で悪化なし)

- **LOW-2 — Content-Security-Policy が未設定 (継続)**: `packages/gui/src/renderer/index.html`。Phase 8 でも未対処。React 経由で文字列はエスケープされるため XSS 経路は依然として無いが、二段目の防御として `default-src 'self'` ベースの CSP を `<meta>` か `onHeadersReceived` で適用するのが望ましい。Vite dev サーバーが `'unsafe-eval'` を要求するため production のみ適用する分岐が必要。
- **LOW-3 — クリップボード ペーストにセーフ ガードが無い (継続 / Phase 8 で部分緩和)**: `packages/gui/src/renderer/components/app/Spreadsheet/Spreadsheet.tsx` の `dispatchPaste`。Phase 8 で **改行を含み得る列 (`Lyrics`) と payload が opaque な列 (`Pictures`) の列方向 broadcast 経路が型レベルで塞がれた**。残るは「短文系列を列選択して 1 値ペースト → 全行 fill」のケース (Phase 4 broadcast 規則の本来動作) で、こちらは Save All で実ファイルへ書き戻されるため、依然として `applyPaste` 前段に「行数 > rows.length」「セル長 > N KiB」の確認ダイアログを挟む価値が残る。優先度は v0.1.0 (Phase 7) 当時より低下。
- **LOW-4 — `onSaveTrack` の許可リスト未実装 (継続)**: `packages/gui/src/main/ipc/onSaveTrack.ts`。Phase 8 では Main / IPC に変更が無いため、`filePath` 経路の数 (dialog / recentFiles / D&D / Open Recent) は v0.1.0 と同じ。許可リストを実装する際は引き続き上記 4 経路すべての成功パスを `Set<string>` (resolved) に追記する設計を検討する。

- [ ] LOW-3: ペースト時のセーフ ガード (行数 / 長さ閾値、確認ダイアログ) を入れたか — Phase 8 で Lyrics / Pictures の最悪ケースは塞がれたが、短文系列の broadcast 誤爆は依然として現実的
- [ ] LOW-4: `onSaveTrack` の `filePath` 許可リストを 4 経路 (dialog / recent / D&D / Open Recent) すべてで満たす形で実装したか
- [ ] `expandDroppedPaths` を将来 fast-glob / fs-extra に置き換える際、本レビューが論拠にしている `lstat` (= symlink を辿らない) と「拡張子 allowlist」の二重ガードが崩れないか確認
- [ ] `mme:menu:setState` を将来「partial patch」に変更する場合は、現在の単純 spread が `__proto__` を素通しさせない構造を維持できるか確認
- [ ] `setupLogger` で `electron-log` の `transports.file.resolvePathFn` をカスタマイズする際、書き込み先が `app.getPath("userData")` 配下に留まるか (パス組み立てに Renderer 由来の値が混じらないか) を必ず確認
- [ ] `setupFatalHandlers` の payload に `Error.cause` を含めるよう拡張する場合は、循環参照対策 (構造化クローンが投げる) と native chain の漏出に注意
- [ ] 依存関係: `electron`, `react`, `@base-ui/react` のメジャー アップデート時に既知 CVE を `npm audit` / GitHub Advisory で再チェック。`electron-log` 等のロガーを将来導入する際は本レビューで「ランタイム依存ゼロ」を前提にしている箇所 (とくに「良い点 11」) を再評価
- [ ] `webPreferences.sandbox: true` 下で preload に新しいモジュールを追加する場合、sandbox の制約 (Node API 不可) に抵触しないか確認
- [ ] `ColumnDefinition.selectable` を「列選択は許可するが broadcast 禁止」のような細粒度制御に拡張する場合、Phase 8 の二者択一 (`column` / `cell-only`) によって型レベルで担保していた「Lyrics / Pictures は列選択不可」の不変条件が崩れないか確認 (新規列を追加する際に明示しなければ `"column"` フォールバックする既定値の妥当性も含めて)
