## サマリー

致命的な脆弱性は **検出されず**。本ブランチの差分は `packages/gui` の Spreadsheet inline editor 周辺の UI イベント ハンドリング (3 ソース ファイル + 1 テスト ファイル) と `packages/gui/package.json` の `version` フィールドのみで、IPC チャネル / ファイル I/O / shell 起動 / 動的コード実行 / 認証 経路のいずれにも触れていない。

差分の中身:

- `SpreadsheetBody.tsx`: 編集中セル (`isEditingCell === true`) の `<td>` に対してのみ `onClick` / `onDoubleClick` を `undefined` にして dispatch を抑止 (input 内クリックでの編集解除バグ修正)
- `useEditableCell.ts`: 既存の `validateTagValue` を再利用する `handleBlur` を新設 (validation OK → `onCommit`、NG → `onCancel`)。Enter ハンドラと完全に同じ commit pipeline を blur からも辿らせるだけの追加
- `EditableCell.tsx`: `<input>` の `onBlur` を `onCancel` から `handleBlur` に差し替え (Excel / Numbers の "click elsewhere → confirm" 挙動への合流)
- `Spreadsheet.test.tsx`: 上記 3 つの挙動を担保する Vitest ケースを追加。`RowSeed` テスト fixture に `language` フィールドを追加 (validation 失敗ケースで `tag.language` の ISO-639 regex を利用)
- `package.json`: `version` を `0.1.0` → `0.5.0`

## 良い点 (Strengths)

v0.0.0 → v0.1.0 で確立した防御線をすべて踏襲。本ブランチの inline editor 改善は構造的にも以下の点でプラス:

1. **`webPreferences.sandbox: true` の維持 (継続)** — `packages/gui/src/main/main.ts`。Renderer サンドボックスの境界線は維持。
2. **`expandDroppedPaths` の三重防御 (継続)** — `lstat` (symlink を辿らない) / 深さ 3 制限 / 拡張子 allowlist は変更なし。
3. **`webUtils.getPathForFile` 経由の D&D path 解決 (継続)** — preload は byte 単位で同一なので Renderer が `File` を偽造して絶対パスを得る経路は引き続き無い。
4. **IPC `MenuAction` の閉じた string union (継続)** — `packages/gui/src/main/ipc/types.ts`。新規 IPC チャネル / action の追加なし。
5. **`shell.openExternal` の URL がハードコード (継続)** — Renderer 由来の URL 流入経路は無い。
6. **ログ転送が出力先プロセス内のみ (継続)** — `onLogForward` / `onFatalReport` は外部 echo back しない。
7. **`mergeSettings` の明示フィールド戦略 (継続)** — `theme` / `locale` は scalar のみ採用。
8. **`resolveLocale` の allowlist (継続)** — `SUPPORTED_LOCALES` (`["en", "ja"]`) との一致比較で safe fallback。
9. **ランタイム依存ゼロ ポリシー (継続)** — 本ブランチでも追加なし。Main は `electron` + `@akabeko/music-metadata-editor`、Renderer は React 系のみ。
10. **`@mme/ipc` 仮想モジュールが型のみ (継続)** — `import type` 限定で TypeScript が elide するため renderer bundle に main の value module は到達しない。
11. **`pnpm-workspace.yaml` の `allowBuilds` map (継続)** — postinstall 実行は明示許可した 4 パッケージのみ。
12. **`useEditableCell.handleBlur` が既存 `validateTagValue` を再利用 (本ブランチ新規)** — `packages/gui/src/renderer/components/app/Spreadsheet/SpreadsheetBody/cells/useEditableCell.ts`。Enter ハンドラが Phase 4 から使ってきたバリデータと同一の `validateTagValue(field, value)` を呼び、`result.ok === false` ケースでは `onCommit` ではなく `onCancel` に分岐させる構造。型レベルでも `validateTagValue` は `string | number | undefined` のみ返すため、blur 経由でも invalid な原文字列が `onCommit` 経由で Main の `onSaveTrack` に到達することはない。
13. **`SpreadsheetBody.tsx` の `<td>` ハンドラ抑止が純粋に防御的 (本ブランチ新規)** — `isEditingCell === true` のとき `onClick` / `onDoubleClick` を `undefined` に渡すだけで、新規 dispatch 経路は作らない。同セル内 input への click が誤って `setEditing(null)` を呼ぶ「リエントリ」を構造的に塞ぐ修正で、攻撃面はむしろ縮小。
14. **`EditableCell.tsx` が `dangerouslySetInnerHTML` を使わない (継続)** — `title={errorMessage ?? undefined}` も React が DOM 属性として安全にセットする経路で XSS 経路無し。
15. **commit pipeline (`commitFromEditor` → `onCommit` → core writer) は変更なし (継続)** — `packages/gui/src/renderer/components/app/Spreadsheet/useSelection.ts` の `commitFromEditor` は本ブランチで触らず、blur からの `onCommit` 呼び出しも Enter からの呼び出しと同じ引数型 / 同じ field 解決経路を辿る。Main 側の信頼境界は変わっていない。

## 所見 (Findings)

### `fix/spreadsheet-editing-cell-click` 差分起因の新規 Finding

**該当なし**。本ブランチは React の DOM イベント ハンドラ配線と既存バリデータの再利用のみで、IPC チャネル / ファイル I/O / 動的コード実行 / shell 起動 / 認証 / 暗号 / ナビゲーション制御のいずれにも触れていない。

検証済の保留 (false positive 判定):

- **blur 経路で commit が走るようになったことで invalid 値が Main に流れる懸念**: `useEditableCell.handleBlur` は `validateTagValue` の `result.ok === false` ケースで `onCancel` に分岐する。validation を通った値だけが `onCommit` に渡る点は Enter ハンドラと同一で、`onCommit` が受け取る型は `string | number | undefined` に限定される。Main 側の `onSaveTrack` パイプラインが想定する型と一致しており、新たな injection 経路は無い。
- **編集中セルへの click が `<td>` ハンドラを通らないことで focus trap になる懸念**: 編集解除は引き続き「フォーカスを失う」「Esc」「Enter (validation OK)」の 3 経路で成立する。`isEditingCell` フラグは React 側 state (`editing`) から導出しており、Renderer 内部のみの判定で外部攻撃者が `isEditingCell` を偽造して dispatch を抑止する経路は無い。
- **`Spreadsheet.test.tsx` の `RowSeed.language` 追加で外部から不正な language コードを差し込まれる懸念**: テスト ファイル (HARD EXCLUSION 11) の話で production code path には到達しない。fixture は `vi.fn()` で受ける `onCommit` を介して assertion するだけで I/O 無し。
- **`package.json` の `version` を `0.1.0` → `0.5.0` に上げたことに伴う signature 検証の懸念**: `private: true` パッケージで npm registry には publish されない。バージョン文字列は electron-builder の installer メタデータと `app.getVersion()` 戻り値に反映されるが、いずれも信頼境界を超えない (ローカルの desktop install 用)。

### v0.1.0 から継続している LOW (`fix/spreadsheet-editing-cell-click` で悪化なし)

- **LOW-2 — Content-Security-Policy が未設定 (継続)**: `packages/gui/src/renderer/index.html`。本ブランチでも未対処。React 経由で文字列はエスケープされるため XSS 経路は依然として無いが、二段目の防御として `default-src 'self'` ベースの CSP を `<meta>` か `onHeadersReceived` で適用するのが望ましい。Vite dev サーバーが `'unsafe-eval'` を要求するため production のみ適用する分岐が必要。
- **LOW-3 — クリップボード ペーストにセーフ ガードが無い (継続)**: `packages/gui/src/renderer/components/app/Spreadsheet/useKeyboard.ts` の `dispatchPaste`。Phase 8 で改行を含む列 (`Lyrics`) と payload が opaque な列 (`Pictures`) の列方向 broadcast 経路は型レベルで塞がれているが、「短文系列を列選択して 1 値ペースト → 全行 fill」のケース (Phase 4 broadcast 規則の本来動作) で、`applyPaste` 前段に「行数 > rows.length」「セル長 > N KiB」の確認ダイアログを挟む価値が引き続き残る。本ブランチで blur commit を導入したことで「うっかり別セルにフォーカスを移して invalid とは限らない値を確定」が起こりやすくなったため、相対的に LOW-3 の優先度はやや上がる。
- **LOW-4 — `onSaveTrack` の許可リスト未実装 (継続)**: `packages/gui/src/main/ipc/onSaveTrack.ts`。本ブランチでは Main / IPC ハンドラに変更が無いため、`filePath` 経路の数 (dialog / recentFiles / D&D / Open Recent) は v0.1.0 と同じ。
