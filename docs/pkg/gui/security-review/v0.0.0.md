## サマリー

致命的な脆弱性は **検出されず**。Phase 6 で追加された **設定永続化 (`packages/gui/src/main/settings/`) と save フロー (`onSaveTrack` / `saveDirtyRows`)** は、いずれも信頼境界 (Renderer = 部分的に信頼、Main = 信頼ドメイン、`userData` = Electron 管理) を踏襲し、新たな攻撃面を持ち込んでいない。特に懸念視していた **prototype pollution と path traversal** は、`mergeSettings` の明示フィールド戦略 + `sanitizeWidths` のフィルター、`app.getPath("userData")` 由来の固定ファイル名で塞がれている。

|  | 件数 |
| --- | --- |
| Critical | 0 |
| High | 0 |
| Medium | 0 |
| Low / Informational | 4 |

LOW は前回 (Phase 4 時点) からの持ち越し 3 件 + Phase 6 の新規 1 件。すべて defense-in-depth または将来的な改善余地で、現フェーズで PR をブロックする要素は無い。

## 良い点 (Strengths)

1. **`contextIsolation: true` + `nodeIntegration: false`** — `packages/gui/src/main/main.ts` で Renderer は Node API に直接触れない。Preload (`contextBridge.exposeInMainWorld`) のみが境界。
2. **IPC 戻り値が常に `IpcResult<T>` で正規化** — `packages/gui/src/main/ipc/types.ts` の `IpcResult` で `try/catch` 不要。Main 側の `toIpcError` が `IpcError` (Plain Object) に詰め替えるため、IPC 越しに stack 情報が漏れない。`onSaveTrack` / `onSetSettings` も同パターンを踏襲。
3. **設定ファイル パスがユーザー入力に汚染されない** — `loadSettingsSync` / `saveSettings` の `userDataDir` は `app.getPath("userData")` (Electron 管理) を `main.ts` で 1 回読むのみ。ファイル名は定数 `"settings.json"`。Renderer 由来のパス断片は `path.join` に流れ込まず、path traversal が成立しない。
4. **`mergeSettings` が prototype pollution に強い** — `version` は `current.version` から固定 (patch 側を捨てる)、`mergeWindow` は `width`/`height`/`maximized` の明示フィールドのみ読む (patch を spread しない)、`mergeColumns` は spread はあるが結果を **`sanitizeWidths` で `typeof === "number" && Number.isFinite && > 0`** に篩い落とすため、`__proto__: { polluted: true }` 型のペイロードは数値ではなくオブジェクトであり弾かれる。`Object.fromEntries` は own data property を作るため、仮に key 名が `__proto__` で値が数値であっても `Object.prototype` を汚染しない。
5. **`loadSettingsSync` の早期排除** — JSON.parse の結果が `null` / 配列 / プリミティブ なら即 `defaultSettings` にフォールバック。誤った top-level 型が `mergeSettings` に到達しない。
6. **Atomic write が symlink を辿らない** — `saveSettings` は `tmp` → `rename` 方式で、`fs.promises.rename` は宛先側の symlink を辿らず置き換える。userData が単一ユーザー領域である desktop モデル下では symlink レースが現実的でない。
7. **`onSaveTrack` の例外正規化** — core の `MmeError` を含む全例外を `toIpcError` 経由で IpcError に変換。`Error.cause` を捨てるため、未処理の native error chain が Renderer に露出しない。
8. **Renderer が `dangerouslySetInnerHTML` / `eval` を一切使用しない** — Phase 6 で追加された `ColumnsMenu` / `SavingDialog` / `Header` / `FileNameCell` も React のテキスト ノード経由で描画。`row.filePath` / `progress.filePath` 等のユーザー入力値も自動エスケープされる。
9. **進捗 IPC は片方向 (`webContents.send`)** — `mme:progress:save` は Main → Renderer の通知のみで、Renderer 側に新しい invoke channel を作らない。`isDestroyed?.()` チェックで race による send 失敗も握りつぶす (情報漏洩を伴わない `console.error` のみ)。

## 所見 (Findings)

### LOW-1 — `webPreferences.sandbox: true` が明示されていない (継続)

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/gui/src/main/main.ts` の `BrowserWindow` 構築箇所 |
| 攻撃シナリオ | Renderer が侵害された場合 (Chromium ゼロデイなど) に Chromium のサンドボックスでさらに 1 段封じ込めたい場面でのリスク低減策が未適用 |

**所見.** Phase 4 時点と同じ。`contextIsolation: true` / `nodeIntegration: false` で Renderer は Node API に到達できないが、`sandbox: true` が明示されていない。Phase 6 で `onSaveTrack` (任意パス書き込み) を本実装したため、Renderer 侵害時の最終防衛ラインの価値が上がった。

**推奨.** `webPreferences.sandbox: true` を追加し、preload (`.cjs` ビルド) との互換を確認。

### LOW-2 — Renderer の Content-Security-Policy が未設定 (継続)

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/gui/src/renderer/index.html`、開発時の Vite HMR / 本番ビルド共通 |
| 攻撃シナリオ | 万一 XSS 経路ができた場合に外部スクリプト読み込みやインライン実行を抑止できない |

**所見.** Phase 4 時点と同じ。Phase 6 でも CSP は導入されていない。React 経由で文字列はエスケープされるため現状の XSS 経路は無いが、将来 `dangerouslySetInnerHTML` や Iframe を入れたときに二段目の防御が無い。

**推奨.** 本番ビルド向けに `Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; font-src 'self' data:` を `<meta>` または `onHeadersReceived` で適用。Vite dev サーバーは `'unsafe-eval'` が必要なため、production のみ適用する分岐を main.ts に入れる。

### LOW-3 — クリップボード ペーストにセーフ ガードが無い (継続)

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/gui/src/renderer/components/app/Spreadsheet/Spreadsheet.tsx` (`dispatchPaste`) |
| 攻撃シナリオ | 別アプリのクリップボード機微情報 (パスワード、トークン) が誤操作で `tag.<field>` に書き込まれ、**Phase 6 で実装された Save All で実際にファイルへ書き戻される** |

**所見.** Phase 6 で `mme:track:save` が動作するため、Phase 4 時点の「ファイルには書かれない」セーフティネットが消えた。誤ペーストがそのまま MP3 / FLAC のタグに焼き付く可能性がある。

**推奨.** `applyPaste` の手前で「行数 > rows.length」「1 セルあたりのバイト長 > N (例: 4 KiB)」のいずれかで `confirm` ダイアログを挟む。短期的には Save All のモーダル ダイアログ内に「直前の paste 件数」を再掲示するなどユーザー側の気付きを補強する案も検討。

### LOW-4 — `onSaveTrack` の許可リスト未実装 (Phase 6 新規 / Phase 4 で持ち越し予告)

| 項目 | 値 |
| --- | --- |
| 重要度 | Low (Informational) |
| 該当 | `packages/gui/src/main/ipc/onSaveTrack.ts` |
| 攻撃シナリオ | Renderer 全体が侵害された場合 (例: 将来導入される modal の脆弱な依存ライブラリ経由)、攻撃者は任意の `filePath` を IPC で渡し、ユーザー権限で書き込み可能な任意ファイル (`~/.ssh/authorized_keys`、シェル設定ファイルなど) の内容を core の `writeMetadata` を経由した「音楽フォーマットとしてパースできる断片 + 残り」で破壊できる。core はパース不能なら `MmeError` を投げるため上書き自体は限定的だが、**書き込み試行で元ファイルが破壊される** ことは fs.writeFile の挙動上ありえる |

**所見.** Phase 4 で予告した通り Phase 6 配線時に対処すべきだったが、本 PR は単一ユーザー デスクトップの脅威モデル (Renderer は user proxy) を前提に「`filePath` 検証なし」のまま実装した。現実的な攻撃成立条件は厳しい (Renderer 侵害 + 任意 IPC 送信能力 + 攻撃対象ファイルに音楽ファイル拡張子 / シグネチャがない場合は writeMetadata が早期失敗するため上書きは限定的) ため、HIGH/MEDIUM ではなく LOW で計上する。

**推奨.** Main 側で「直近の `dialog.showOpenDialog` で返却したパス集合 + Open Recent から再ロードしたパス集合」を `Set<string>` に保持し、`onSaveTrack` 入口で `if (!openedPaths.has(path.resolve(request.filePath))) return error`。`path.resolve` で正規化してから比較。Phase 7 で `Open Recent` メニューを実装するタイミングと合わせて入れるのが自然。

## 次回更新時のチェックリスト

gui のバージョンを v0.1.0 等に上げる PR では **新しい `v0.x.y.md` を作成** し、以下を再評価:

- [ ] LOW-1: `sandbox: true` を main.ts に追加したか / preload (.cjs) との両立を確認したか
- [ ] LOW-2: production ビルドの CSP を導入したか (Vite dev/prod の分岐含む)
- [ ] LOW-3: ペースト時のセーフ ガード (行数 / 長さ閾値、確認ダイアログ) を入れたか — **Phase 6 で save 配線が完了したため優先度が上がった**
- [ ] LOW-4: `onSaveTrack` の `filePath` 許可リスト (dialog 経由で開いたパスのみ) を実装したか
- [ ] `mergeSettings` を将来 lodash 等の汎用 deep-merge ライブラリに差し替える際は、本レビューが論拠としている **明示フィールド戦略 + `sanitizeWidths` フィルター** が崩れないか必ず確認 (汎用 deep-merge は prototype pollution 経路を作りやすい)
- [ ] `saveSettings` が `tmp` ファイルを `<userData>/settings.json.tmp` 固定にしている点 — マルチプロセスの Electron 構成 (将来導入される場合) に切り替える際は、tmp ファイルにプロセス ID を含める等の競合回避を検討
- [ ] `onSaveTrack` が原子的書き込みでない (現状 `fs.writeFile` で直接 `filePath` 上書き) ため、書き込み中のクラッシュで元ファイルが破壊される可能性。core 側で atomic write が実装されたら gui 側は不要だが、現状は未対応 — Phase 7 以降で再評価
- [ ] Phase 7 で `Open Recent` メニュー / native menu accelerator を入れる際、Renderer 経由で `filePath` を再オープンする経路が増えるため LOW-4 の許可リストと整合させる
- [ ] 依存関係: `electron`, `react`, `@base-ui/react` のメジャー アップデート時に既知 CVE を `npm audit` / GitHub Advisory で再チェック
